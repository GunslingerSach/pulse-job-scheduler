# API Documentation

Interactive, always-in-sync docs are auto-generated by FastAPI and served at
**`/docs`** (Swagger UI) and **`/redoc`** when the server is running. This
document is a human-readable index of the same API.

Base URL (local dev): `http://localhost:8000`

## Authentication

Two schemes, used by different callers:

| Caller | Scheme | Header |
|---|---|---|
| Dashboard / human users | JWT bearer token | `Authorization: Bearer <token>` |
| Worker processes | Project API key | `x-api-key: <project.api_key>` |

Get a JWT via `POST /api/v1/auth/register` or `/login`. Get a project's API
key from the `ProjectOut.api_key` field returned by `GET /api/v1/projects`.

## Queues

All under `/api/v1/projects/{project_id}/queues`.

| Method | Path | Description |
|---|---|---|
| POST | `` | Create a queue; optionally embed a `retry_policy` object. |
| GET | `` | List queues for the project. |
| PATCH | `/{queue_id}` | Update `priority`, `max_concurrency`, `is_paused`. |
| POST | `/{queue_id}/pause` | Stop new claims against this queue. |
| POST | `/{queue_id}/resume` | Resume claiming. |
| GET | `/{queue_id}/stats` | Live counts by status, avg duration, last-hour throughput. |

**Example — create a queue with exponential backoff:**
```bash
curl -X POST $BASE/api/v1/projects/$PROJECT_ID/queues \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{
        "name": "emails",
        "priority": 5,
        "max_concurrency": 10,
        "retry_policy": {
          "name": "email-retry",
          "strategy": "exponential",
          "max_retries": 4,
          "base_delay_seconds": 5,
          "max_delay_seconds": 3600,
          "multiplier": 2.0
        }
      }'
```

## Jobs

All under `/api/v1/projects/{project_id}/queues/{queue_id}/jobs`.

| Method | Path | Description |
|---|---|---|
| POST | `` | Create a job. `type` is one of `immediate\|delayed\|scheduled\|recurring\|batch`. |
| POST | `/batch` | Create many jobs atomically under one `batch_id`. |
| GET | `` | List jobs. Query params: `status`, `job_type`, `page`, `page_size` (pagination + filtering). |
| GET | `/{job_id}` | Job detail. |
| GET | `/{job_id}/executions` | Full attempt history for a job. |
| GET | `/{job_id}/logs` | Structured log lines for a job. |
| POST | `/{job_id}/cancel` | Cancel a non-terminal job. |
| POST | `/{job_id}/retry` | Manually requeue a `failed`/`dead_letter` job. |

**Job type → required fields:**
- `immediate` — nothing extra required; claimable right away.
- `delayed` / `scheduled` — requires `run_at` (ISO 8601); stays `SCHEDULED` until then.
- `recurring` — requires `cron_expression`; creates a template that materializes each occurrence as a `scheduled` job.
- `batch` — use `POST /batch` with a `jobs: [...]` array; all created rows share one `batch_id`.

**Example — create a recurring job:**
```bash
curl -X POST $BASE/api/v1/projects/$PID/queues/$QID/jobs \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{
        "name": "generate_report",
        "type": "recurring",
        "cron_expression": "0 * * * *",
        "payload": {"report_id": "daily-usage"}
      }'
```

**Idempotency:** pass `idempotency_key` on job creation; a second `POST` with
the same key (within the same queue) returns the original job instead of
creating a duplicate.

## Workers

All under `/api/v1/workers`, authenticated with `x-api-key`. These are the
endpoints the worker process (`app/worker/runner.py`) calls automatically —
you generally don't call these by hand except `GET /api/v1/workers` for
dashboard/monitoring purposes.

| Method | Path | Description |
|---|---|---|
| POST | `/register` | Register a new worker process, get back a `worker_id`. |
| POST | `/{worker_id}/heartbeat` | Report liveness + current load. |
| POST | `/{worker_id}/drain` | Mark draining (graceful shutdown in progress). |
| POST | `/{worker_id}/shutdown` | Mark offline. |
| GET | `` | List all workers for the project (used by the dashboard). |
| POST | `/{worker_id}/claim` | Atomically claim up to `max_jobs` runnable jobs. |
| POST | `/{worker_id}/jobs/{job_id}/start` | Mark a claimed job `RUNNING`, open a `JobExecution`. |
| POST | `/{worker_id}/jobs/{job_id}/complete` | Record success + result. |
| POST | `/{worker_id}/jobs/{job_id}/fail` | Record failure; triggers retry-or-DLQ transition. |

## Dead Letter Queue

All under `/api/v1/projects/{project_id}/dead-letter`.

| Method | Path | Description |
|---|---|---|
| GET | `` | List DLQ entries across all the project's queues. |
| POST | `/{dlq_id}/replay` | Re-enqueue a DLQ entry as a fresh job (retry_count reset to 0). |

## Stats

All under `/api/v1/projects/{project_id}/stats`.

| Method | Path | Description |
|---|---|---|
| GET | `/overview` | Project-wide counts by status + worker counts, for dashboard cards. |
| GET | `/throughput?hours=24` | Completed/failed counts bucketed by hour, for the throughput chart. |

## Errors

Errors are returned as `{"detail": "..."}` with a standard HTTP status code
(400 validation, 401 unauthenticated, 403 unauthorized, 404 not found, 422
request-schema validation, 500 unhandled — logged server-side and never leaks
internals to the client).
